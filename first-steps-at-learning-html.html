First Steps at Learning HTML

Introduction

When I was an intern back in 2021, by boss once called frontend development "coloring pixels". For some reason, that didn't really resonate with me. I was learning C# for my internship, and I liked the robustness of OOP. I liked creating objects and then having those objects interact with each other. It was all very structured and logical. Hearing the phrase "coloring pixels" made me think that frontend development was more of a classic art, like painting, and less of a structured way of having different elements on the page interact with each other. I thought you had to meticulously put every element exactly where it's supposed to go when creating a webpage. If you wanted to add a new element to the page, you'd have to color the pixels of the screen in the exact way that would allow the element to be made.

Now, I knew that my boss was making a joke at the time, but I didn't know how frontend development actually worked. I knew about HTML, CSS, and JavaScript, but I didn't know how they worked. Whenever I was tasked with having something appear on the screen, I would just fiddle around with <div>s and use the already existing TypeScript functions to create the desired elements. Only in 2024, when I had worked as a junior developer for 2 years and put more focus on learning and personal development, did I start learning the ins and outs of frontend development. In this journey, I figured out that frontend development is more than "coloring pixels", and that it's actually very structured. Also, with the European Accessibility Act taking effect in June of 2025, I figured now would be a great time to start learning how I can make the web more accessible.

More than coloring pixels

I once spent a couple of weeks refactoring some old code that I wrote as an intern a couple years prior. The code was sloppy and hard to follow, it was kind of like how an intern would write their code. I fixed it up with all the new shiny techniques that I had learned and made it robust and efficient, with clear comments and documentation. At the end of the couple of weeks, I was done, and the feature worked exactly the same as it did before the refactor. I hadn't changed the actual user interface, it was still the same buttons in the same order and the results were displayed in the same way. I spent multiple weeks working really hard only for the end user to not notice any difference.

This experience made me realize that development is more than just "making it work". It's about writing readable and maintainable code so that other developers can more easily debug it. If you have a massive function that is hundreds of lines long, you're better off breaking it up in many smaller functions. That may not change the logic, the functions might still do the exact same thing combined as what the initial massive function did, but at least now you have modular code that is way easier to debug, you're not just writing code for the end user. You're also writing code for yourself and for anyone else that works on the same codebase.

I always knew this was the case for writing code in programming languages, but I only recently realized that this concept also applies to writing HTML. The "coloring pixels" concept made me think that HTML was just a means of making your content appear on the screen in a way that looks good. HTML however, is much more than that.

Semantics, not styling

As a (primarily) backend developer with some experience with frontend development (TypeScript and React), my time spent actually constructing HTML has been limited so far. Usually when I was tasked to have something appear on the screen, I would call preexisting functions or copy and paste the HTML from a different element. If I didn't like the position of an element, I would mess around with the HTML. Maybe I'd wrap some elements in a <div> or use a <br/> tag to separate two paragraphs. I was using HTML as a means of styling the page.

I didn't understand how CSS worked back then, and messing around with the HTML got me the desired results, so I would be satisfied and move on. Of course, I would have no idea of the implications this would have on how the browser actually reads and understands the code. The implications being semantics.

HTML doesn't just look at the document to decide which piece of content to put where and how to style each piece of content, that's mostly the job of CSS. HTML looks at the document to decide what each piece of content actually means. Is it a header? A paragraph? A table? Once the browser knows what type of content something is, it can show it to the CSS in order to style it. It also knows how to treat the piece of content in other ways. You could use JavaScript to automatically create a table of contents of all the <h1>, <h2>, etc. tags. Screenreaders also know how to read each piece of content to the user. Is emphasis used? Define it with the <em> tag. Is something important? Use the <strong> tag. Note that these are different from italics and bold tags, which use <i> and <b> respectively. <i> and <b> are just simple styling tags, which you can use to style a piece of text however you want with CSS. In contrast, the <em> and <strong> tags hold actual meaning. They are used to tell the browser that the pieces of text within those tags should be conveyed to the user in a certain way. In a practical example, a screenreader will use actual emphasis in their voice when reading a piece of text wrapped in an <em> tag, meaning you can hear it when it's being read to you. A person with a visual disability may not be able to see that the text is written in italics, but they will hear the emphasis through the screen reader. This means that marking up text is more than just showing the user some text, it is conveying the desired meaning to the user. This can be done through multiple channels, like seeing and hearing.

The job of HTML is to provide a structure to your webpage and to define the different elements that you use. It doesn't matter what the results actually look like on your screen, you can always change that with CSS. It matters that correct meaning is conveyed to the browser so in turn it can convey that meaning to the user in whatever way is preffered.

Wrapping a header in a <header> tag doesn't make it a neat bar at the top of the page. I've made the decision to not use CSS in this article just yet. That is mainly because I want to get used to the HTML semantics first. I also know that it can be very tempting to use the wrong tag just to change the styling of an element. Maybe an <h2> in a certain style is a little too big, and I'd rather use an <h3> element. If that were to happen, I would be sacrificing semantic meaning over the styling of my page, which means the value of HTML is lost.

A browser should be able to read a webpage as well as possible, providing the proper semantics is essential in this process. Once the HTML code is valid and semantically sound, I can make the page look like anything I want with proper CSS.

Overdoing it

You may have noticed that every time I write the letters HTML and CSS in this article, the browser provides a tooltip with the expansion of the abbreviation. It does this because I have wrapped the abbreviations in an <abbr> tag. Now, you might say that the definition of an abbreviation only has to be specified once, perhaps in the title of this article. I think, however, that specifying the definition of an abbreviation every time has value, even if it seems like overkill to do it this much.

In my opinion, "overdoing it" doesn't have to be a thing. In terms of HTML code, you can never tell the receiver of the code too much in terms of semantics. You might choose to hide certain tags and attributes with CSS, but having the browser know what each piece of content means and represents, means that the browser can always display it in the way that you want and that the browser can do with it what it needs to do with it. You could hide dirty HTML behind a layer of CSS and make the site appear exactly the same as a site with proper HTML, but in doing so you forfeit meaning and semantics, which can lead to discrepancies in the future or accessibility issues.

It's the same opinion I have when it comes to writing comments and documentation in code. You can figure out what many variables and functions are for by just reading the code itself, but adding comments ensures that even the most inexperienced (or tired!) developer can make up what's going on.

In the end, the receiver of the HTML code just needs to know what each piece of content means. The browser can guess, sometimes it can guess pretty well, but providing it with the right context ensures that it doesn't have to guess.

Underdoing it

While you could spend time learning all the different HTML tags and what they're used for, you could also just wrap everything in a <div>. You could define your own tags by giving each <div> a corresponding class name and then styling them all with CSS. If you do that, you only have to mark up content that you think needs to be marked up. You could have <div>s in <div>s in <div>s, and as long as you write good CSS, your page could look exactly the same as one where proper HTML tags are used. So, what's the point?

The point is, as I've touched upon extensively in this article, semantics. The browser has no idea what your <div> is supposed to represent, and maybe in 5 years, you don't either. As I've explained, the point of HTML is to provide meaning and structure to a webpage. If you don't like the way a certain tag looks, change the styling in CSS, don't forfeit meaning by changing the HTML.

I'm not saying that you should never use a <div> or a <span>, they can be quite useful for wrapping multiple pieces of content or to style a certain piece of text. I am saying that you should always consider if there is another tag for your use case that you could use to give your content proper meaning.

Beyond HTML; CSS and JavaScript

In this article, I've talked mostly about HTML. HTML is only a part of what makes a webpage a webpage however. CSS can be used to style the page and JavaScript is used as a scripting language to interact with the HTML. There's much more to it that just creating simple static websites. I haven't even talked about things like <form>s in this article, which is something that many webpages have.

As is often the case with programming projects, different languages offer different advantages. In the case of web development, the three aforementioned languages are used together to create a user-friendly experience on the web.

At the time of writing this, I know very little about CSS, and although I have experience in TypeScript, I bet there is also still a lot that I can learn about JavaScript. Perhaps I will write about my experience learning those languages on this blog as well.

Conclusion

HTML is the language of the web. It is a means of communicating between webpage developers and their users, whoever they may be. It can be used to aid those who have trouble reading or navigating a page based on purely styling, and it can provide rules and guidelines for developers in order to write in a somewhat common tongue.

Using HTML in the right way can make the web more robust and accessible for many different people. Using it in the wrong way could create horrible experiences for some, which is not only bad for them, but also for your website's traffic. I therefore hope that most developers strive for writing proper HTML, not sacrificing semantics for an easier time creating a website.

There are plenty of HTML tags that I still need to learn, and I'm sure I'm using some of the ones that I do know in a wrong way. Through trial and error I hope to learn all I can to be able to make webpages that use proper HTML in order to make the web a better and more accessible place for everyone.
