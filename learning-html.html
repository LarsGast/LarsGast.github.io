Learning HTML

Introduction

When I was an intern back in 2021, by boss once called frontend development "coloring pixels". For some reason, that didn't really resonate with me. I was learning C# for my internship, and I liked the robustness of OOP. I liked creating objects and then having those objects interact with each other. It was all very structured and logical. Hearing the phrase "coloring pixels" made me think that frontend development was more of a classic art, like painting, and less of a structured way of having different elements on the page interact with each other. I thought you had to meticulously put every element exactly where it's supposed to go when creating a webpage. If you wanted to add a new element to the page, you'd have to color the pixels of the screen in the exact way that would allow the element to be made.

Now, I knew that my boss was making a joke at the time, but I didn't know how frontend development actually worked. I knew about HTML, CSS, and JavaScript, but I didn't know how they worked. Whenever I was tasked with having something appear on the screen, I would just fiddle around with divs and use the already existing TypeScript functions to create the desired elements. Only in 2024, when I had worked as a junior developer for 2 years and put more focus on learning and personal development, did I start learning the ins and outs of frontend development. In this journey, I figured out that frontend development is more than "coloring pixels", and that it's actually very structured. Also, with the European Accessibility Act taking effect on June 28 2025, I figured now would be a great time to start learning how I can make the web more accessible.

More than coloring pixels

Semantics, not styling

As a primarily backend developer with some experience with frontend development (TypeScript and React), my time spent actually constructing HTML has been limited so far. Usually when I'm tasked to have something appear on the screen, I call preexisting functions or I copy and paste the HTML from a different element. If I don't like the position of an element, I mess around with the HTML, or maybe I wrap some elements in a div, or use a <br/> tag to separate two paragraphs. I was using HTML as a means of styling the page.

I didn't understand how CSS worked back then, and messing around with the HTML got me the desired results, so I would be satisfied and move on. Of course, I would have no idea of the implications this would have on how the browser actually reads and understands the code. The implications being semantics.

HTML doesn't just look at the document to decide which piece of content to put where and how to style each piece of content, that's mostly the job of CSS. HTML looks at the document to decide what each piece of content actually means. Is it a header? A paragraph? A table? Once the browser knows what type of content something is, it can show it to the CSS in order to style it. It also knows how to treat the piece of content in other ways. You could use JavaScript to automatically create a table of contents of all the <h1>, <h2>, etc. tags. Screenreaders also know how to read each piece of content to the user. Is emphasis used? Define it with the <em> tag. Is something important? Use the <strong> tag. Note that these are different from italics and bold tags, which use <i> and <b> respectively. <i> and <b> are just simple styling tags, which you can use to style a piece of text however you want with CSS. In contrast, the <em> and <strong> tags hold actual meaning. They are used to tell the browser that the pieces of text within those tags should be conveyed to the user in a certain way. In a practical example, a screenreader will use actual emphasis in their voice when reading a piece of text wrapped in an <em> tag, meaning you can hear it when it's being read to you. A person with a visual disability may not be able to see that the text is written in italics, but they will hear the emphasis through the screen reader. This means that marking up text is more than just showing the user some text, it is conveying the desired meaning to the user. This can be done through multiple channels, like seeing and hearing.
